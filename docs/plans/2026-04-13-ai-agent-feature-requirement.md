## 1. 项目背景

### 1.1 产品简介

CorineKit Pix2Real 是一个基于 ComfyUI 的图像生成与处理应用，面向二次元图像创作场景，提供从文生图到风格转换、精修放大等全流程能力。

### 1.2 技术架构

- **前端**：React + TypeScript + Vite
- **后端**：Express + TypeScript
- **通信**：REST API + WebSocket（实时进度推送）
- **AI 能力**：Grok API（已集成，用于提示词处理和图像反推）
- **图像引擎**：ComfyUI（本地部署）

### 1.3 现有能力

| 能力模块 | 说明 |
|---------|------|
| 工作流系统 | 10+ 图像/视频处理工作流（二次元转真人、真人精修、ZIT快出、高清重绘、视频生成等） |
| 模型选择 | 支持 Checkpoint 模型下拉选择，已维护模型元数据（昵称、分类、缩略图） |
| LoRA 组合 | 支持最多 5 个 LoRA 同时启用，每个 LoRA 可配置触发词和权重 |
| 提示词助手 | 基于 Grok API 的提示词转换和扩写功能 |
| 会话管理 | 支持会话持久化、输出文件管理 |

## 2. 功能目标

### 2.1 核心价值

通过引入 AI Agent，让用户以自然语言对话的方式完成图像生成和处理任务，降低操作门槛，提升使用效率。

### 2.2 核心能力

1. **自然语言理解**：理解用户的图像生成/处理意图
2. **智能模型推荐**：根据需求自动选择基础模型和 LoRA 组合
3. **工作流编排**：自动规划并执行多步骤任务链
4. **对话式交互**：通过聊天界面与用户确认、调整、反馈
5. **图片上传与处理**：支持用户在对话中上传图片，进行 Inpainting 等图像处理操作
6. **可扩展能力注册**：支持后续新增工作流、Tab、处理能力时无需修改 Agent 核心逻辑

### 2.3 预期收益

- 用户操作步骤减少 50% 以上
- 新用户上手时间缩短
- 复杂多步任务自动化执行

## 3. 功能需求

### 3.1 对话界面

#### 3.1.1 界面布局

- 以独立 Tab 或悬浮面板形式存在
- 聊天式交互界面，支持消息历史滚动
- 输入框支持多行文本
- 输入框支持图片上传（点击或拖拽）
- 集成到现有 UI 风格体系

#### 3.1.2 消息展示

- 用户消息：右对齐气泡
- 用户图片消息：显示缩略图，支持点击放大预览
- Agent 消息：左对齐气泡，支持 Markdown 渲染
- 任务卡片：展示推荐方案（模型、LoRA、参数）
- 进度指示：展示当前任务执行状态和进度百分比
- 结果展示：图片缩略图 + 操作按钮（下载、继续处理）
- Inpainting 结果对比：原图 vs 编辑后图片并排展示

### 3.2 意图理解与模型推荐

#### 3.2.1 意图解析维度

Agent 需从用户输入中提取以下信息：

**文生图场景：**

| 维度 | 说明 | 示例 |
|------|------|------|
| 角色 | 具体人物角色名 | 菲谢尔、安琪拉、艾莲·乔 |
| 姿势 | 特定姿势关键词 | 壁尻、战败、足交 |
| 表情 | 面部表情 | 阿嘿颜、嫌弃脸、厌恶脸 |
| 风格 | 画面风格 | 赛博朋克、写实、动漫 |
| 质量 | 质量要求 | 快速出图、高质量、高清 |
| 数量 | 生成数量 | 生成3张、批量生成 |
| 后续处理 | 是否需要后续工作流 | 放大、精修、转真人 |

**图片处理场景（用户上传图片时）：**

| 维度 | 说明 | 示例 |
|------|------|------|
| 输入图片 | 用户上传的图片 | 聊天中上传的图片 |
| 处理类型 | 图片处理方式 | 区域编辑、风格转换、放大、精修 |
| 编辑区域 | Inpainting 的目标区域 | 衣服、背景、特定部位 |
| 编辑目标 | 编辑后想要的效果 | 换成比基尼、去掉衣服、换个表情 |
| 蒙版来源 | 如何确定编辑区域 | 自动识别、手动涂抹、语义描述 |

#### 3.2.2 图片处理意图识别

当用户上传图片时，Agent 需判断：

1. **处理类型识别**
   - 风格转换："把这张转成真人风格" → 二次元转真人
   - 区域编辑："把衣服换成比基尼" → Inpainting
   - 质量提升："把这张放大清晰一点" → 高清放大
   - 面部处理："换个脸" → 黑兽换脸

2. **编辑区域理解**（Inpainting 场景）
   - 语义理解："把衣服换成..." → 自动识别衣物区域
   - 部位指定："把裙子换成裤子" → 自动识别裙子
   - 全局修改："整体美化一下" → 全图精修

3. **自动蒙版生成**
   - 支持基于语义描述自动生成蒙版
   - 支持引导用户手动涂抹蒙版
   - 支持 AI 自动识别编辑目标（如人物、衣物、背景等）

#### 3.2.3 模型推荐逻辑

**基础模型选择规则：**

| 模型系列 | 适用场景 |
|---------|---------|
| 光辉系列 | 通用动漫风格，适合二次元角色、插画 |
| PONY 系列 | 特定风格需求 |

**LoRA 组合推荐规则：**

- 角色 LoRA + 姿势 LoRA + 风格/质量 LoRA 可组合使用
- 同类型 LoRA 避免重复选择（如两个姿势 LoRA）
- 自动注入所选 LoRA 的触发词到提示词
- 提供推荐强度（strength）建议

#### 3.2.4 参数推荐

- 比例：根据内容类型推荐（角色用竖图、风景用横图）
- 步数：根据质量要求推荐（快速 20 步、高质量 35+ 步）
- CFG：根据模型特性推荐默认值
- 后续工作流：根据"高质量"关键词自动建议高清放大

### 3.3 工作流编排与执行

#### 3.3.1 单任务执行

- 解析意图 → 推荐方案 → 用户确认 → 执行 → 展示结果
- 支持用户修改推荐方案后再执行

#### 3.3.2 多任务编排

- 支持顺序执行（如：生成 → 高清放大）
- 前置任务完成后自动触发后续任务
- 实时推送每个任务的状态和进度
- 支持中途取消

#### 3.3.3 错误处理

- 执行失败时向用户反馈错误原因
- 提供重试选项
- 支持回退到上一步

### 3.4 上下文记忆

#### 3.4.1 会话级记忆

- 记住当前会话的对话历史
- 支持指代消解（如："用刚才的角色换个姿势"）
- 记住已生成的图片，支持后续引用

#### 3.4.2 用户偏好记忆（可选）

- 记住用户常用的模型和 LoRA
- 记住用户的参数偏好

### 3.5 图片上传与 Inpainting

#### 3.5.1 图片上传功能

- 支持在对话中上传图片（点击选择或拖拽）
- 支持粘贴截图直接发送
- 上传后显示图片预览
- 支持上传多张图片（如换脸场景需要源图和目标图）

#### 3.5.2 Inpainting 交互流程

**方式一：语义描述 + 自动蒙版**
```
用户：[上传图片] 把衣服换成比基尼
Agent：识别编辑目标(衣服) → 自动生成蒙版 → 展示蒙版预览 → 用户确认 → 执行
```

**方式二：手动涂抹蒙版**
```
用户：[上传图片] 帮我编辑一下这张图
Agent：请问您想编辑哪个区域？[引导用户涂抹蒙版]
用户：[涂抹蒙版] 换成比基尼
Agent：执行 Inpainting
```

**方式三：选择预设区域**
```
用户：[上传图片] 把背景换掉
Agent：检测到可编辑区域：[人物] [衣服] [背景] [全部]
用户：[选择背景]
Agent：执行背景替换
```

#### 3.5.3 自动蒙版生成

- 集成现有的自动识别功能（Tab 10 区域编辑已有能力）
- 支持基于语义描述的智能蒙版生成
- 支持 SAM (Segment Anything Model) 或类似分割模型

#### 3.5.4 多图处理场景

| 场景 | 输入要求 | 交互方式 |
|------|---------|---------|
| 单图编辑 | 1张图片 + 文字描述 | 自动识别编辑区域或引导涂抹 |
| 换脸 | 2张图片（源脸 + 目标图） | 引导用户指定哪张是源脸 |
| 风格转换 | 1张图片 + 风格描述 | 直接执行转换 |
| 图生图 | 1张图片 + 修改描述 | 根据描述强度调整重绘程度 |

## 4. 可扩展性设计

### 4.1 设计目标

Agent 系统需支持后续扩展，包括但不限于：
- 新增工作流（Tab）
- 新增图像处理能力
- 新增模型/LoRA
- 新增意图类型

**核心原则：新增能力时无需修改 Agent 核心逻辑，只需注册能力描述。**

### 4.2 能力注册机制

#### 4.2.1 工作流能力描述

每个工作流需提供一份能力描述文件，供 Agent 动态加载：

```yaml
# 示例：workflow-registry/0-anime-to-real.yaml
id: 0
name: 二次元转真人
type: style_transfer
description: 将动漫风格的图片转换为写实真人风格
input:
  - type: image
    required: true
    description: 待转换的动漫图片
  - type: text
    name: prompt
    required: false
    description: 额外的风格描述
output:
  - type: image
    description: 转换后的真人风格图片
triggers:
  keywords:
    - 二次元转真人
    - 动漫转真人
    - 转成真人
    - anime to real
  patterns:
    - "把这张.*转成真人"
    - ".*转真人风格"
compatible_with: []  # 可与其他工作流串联
```

#### 4.2.2 工作流注册表

系统维护一份工作流注册表，Agent 启动时自动加载：

```json
{
  "workflows": [
    { "id": 0, "name": "二次元转真人", "type": "style_transfer", ... },
    { "id": 7, "name": "快速出图", "type": "text2img", ... },
    { "id": 10, "name": "区域编辑", "type": "inpainting", ... }
  ]
}
```

### 4.3 新增工作流接入流程

当需要新增工作流时，只需：

1. **实现工作流**：创建 ComfyUI 工作流 JSON 文件
2. **创建适配器**：参考现有 Adapter 模式实现参数映射
3. **注册能力描述**：创建 YAML 文件描述工作流能力
4. **Agent 自动识别**：无需修改 Agent 核心代码

### 4.4 模型/LoRA 元数据扩展

当新增模型或 LoRA 时：

1. 添加模型文件到 ComfyUI models 目录
2. 更新 `metadata.json`，补充描述和标签
3. Agent 自动识别新的模型/LoRA

### 4.5 意图类型扩展

支持后续新增意图类型（如视频生成、批量处理等）：

- 意图处理器采用插件化设计
- 新增意图类型只需实现处理器接口并注册
- Agent 根据用户输入自动路由到对应处理器

## 5. 数据需求

### 5.1 现有数据

项目已维护 `model_meta/metadata.json`，包含：

| 字段 | 说明 | 示例 |
|------|------|------|
| nickname | 中文昵称 | 菲谢尔(原神) |
| category | 分类 | 角色、姿势、表情、风格 |
| triggerWords | 触发词 | gcs_fischl, green eyes... |
| thumbnail | 缩略图文件名 | xxx.png |

### 5.2 需要补充的数据

#### 5.2.1 基础模型描述

| 字段 | 说明 | 必要性 |
|------|------|--------|
| description | 模型风格一句话描述 | 必需 |
| style | 风格标签数组（anime/realistic/2.5D） | 必需 |
| suitableFor | 适用场景数组（角色/风景/概念图） | 推荐 |
| defaultParams | 推荐默认参数（steps/cfg/sampler） | 推荐 |

#### 5.2.2 LoRA 语义标签

| 字段 | 说明 | 必要性 |
|------|------|--------|
| description | LoRA 用途一句话描述 | 必需 |
| tags | 语义标签数组 | 必需 |
| keywords | 匹配关键词数组 | 必需 |
| compatibleModels | 兼容的基础模型系列 | 推荐 |
| recommendedStrength | 推荐强度值 | 推荐 |
| nsfw | 是否为成人内容 | 推荐 |

#### 5.2.3 关键词映射表

用于快速匹配用户输入：

```
"菲谢尔" → ["角色\\【光辉】【角色】菲谢尔.safetensors"]
"壁尻" → ["姿势\\【光辉】【姿势】壁尻A.safetensors", "姿势\\【光辉】【姿势】壁尻B.safetensors"]
"阿嘿颜" → ["表情\\【光辉】【表情】阿嘿颜.safetensors"]
```

### 5.3 数据维护方式

- **初期**：手动补充核心模型/LoRA 的描述和标签
- **中期**：编写脚本，使用 LLM 批量生成描述
- **长期**：基于用户反馈自动优化
